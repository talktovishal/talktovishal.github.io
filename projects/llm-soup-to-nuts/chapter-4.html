Learning by doing

Tokenization: how text becomes the pieces a model reads.

Every chapter so far quietly split text into pieces before doing anything else. Chapter 1 just cut on spaces. Real models do something smarter, and that choice shapes cost, speed, memory, and even fairness. This chapter is about that first, easy-to-overlook step.

Here is the thing to hold onto: a neural model never sees your letters. It sees token IDs, plain integers that index into its embedding table. So before any math happens, some tokenizer has to chop your text into known pieces and look up their numbers. Pick the pieces badly and a common word becomes three tokens; pick them well and a rare word still spells out from familiar parts.

Guiding question: if the word unbelievable never appeared in training, how can the model still handle it? The answer is that it was never stored as one piece. It was stored as parts the model has seen many times.

1. Why tokenization matters

The same sentence can be many different lengths.

Chapter 1 used a simple tokenizer so we could count words. That was fine for counting, but it hides a real decision. Should the unit be a whole word, a single character, a raw byte, or a subword chunk in between? Each answer changes how many tokens a sentence becomes, and token count is what you pay for in money, memory, and latency.

Whole words are easy to read, but rare words have no entry and break. Characters and bytes can spell anything, but they make long sequences that are slow and expensive. Subwords are the middle path used by almost every modern model: keep frequent pieces whole, split rarer words into reusable parts. Type into the box and watch the four views disagree on the very same text.

There is a useful little budget hiding here. If a prompt has T tokens, a Transformer attention layer compares tokens to tokens, so the attention score table has about T * T cells. Doubling the token count does not merely double that table; it makes it roughly four times larger. The tokenizer also chooses the size of the embedding table: with V vocabulary pieces and d numbers per vector, the first table has V * d learned numbers. Tokenization is therefore a tradeoff: a larger vocabulary can shorten sequences, but it also creates more rows the model has to learn.

Token cost by hand

Imagine one sentence becomes 12 tokens with a good subword tokenizer, but 20 tokens with a byte-heavy tokenizer. The text looks the same to you, but the Transformer does more pairwise comparisons.

attention cells with 12 tokens = 12 * 12 = 144 attention cells with 20 tokens = 20 * 20 = 400 400 / 144 = 2.8x as many token-to-token scores

This is why a word that splits into many pieces has a real token tax: it can cost more memory, more time, and more money even before the model starts reasoning about meaning.

2. Tokenizer algorithms

Let the tokenizer learn its pieces from real text.

A hand-written splitter is easy, but modern tokenizers are trained. They scan a corpus and decide which pieces are worth a slot in the vocabulary. That is why frequent chunks stay short while rare words can still be spelled from smaller, known parts. The tokenizer below trains on the corpus you picked above, so the merges are genuine.

This lab is small, but real. BPE repeatedly merges the most frequent neighboring pair. WordPiece-style encoding greedily takes the longest known piece. Unigram-style encoding picks the most likely segmentation from a learned set of pieces. Byte-level encoding skips training and uses the raw UTF-8 bytes. Continuation pieces are shown with a ## prefix.

Two tokenizer algorithms by hand

BPE is a counting algorithm. Start with characters, count adjacent pairs, merge the most frequent pair, then count again. On the tiny corpus hug hug hug hugs hugging hugging, the pairs h u and u g both appear 6 times, so the first merge can turn h + u into hu.

score_BPE(pair) = count(pair in the training corpus) first merge: h + u -> hu because count(h u) = 6 next useful merge: hu + g -> hug, so hug becomes one reusable piece

Unigram thinks differently. It keeps many possible pieces and asks which segmentation has the best total probability. In practice it uses dynamic programming; by hand, you can read it as "choose the pieces with the smaller surprise."

score(segmentation) = sum(-ln p(piece)) un + believ + able: 3.22 + 4.61 + 3.00 = 10.83 nats unbelievable as one rare piece: -ln(0.000001) = 13.82 nats

Lower score wins, so this toy Unigram model prefers the familiar parts. BPE asks "what pair was frequent?"; Unigram asks "what full spelling is most probable?"

3. Tokenization edge cases

Always inspect what the model actually receives.

Edge cases are not trivia. They are where hidden assumptions show up. Multilingual text, code, whitespace, numbers, emoji, rare names, and domain vocabulary can all stretch or distort the token stream. The same algorithm you just trained handles each case below.

Pick a case and inspect the output. Sometimes the visible text and the tokenized input are not as similar as they look, and that gap can affect cost, evidence, safety, and fairness. The hidden-text case is a small preview of prompt-injection risks we will revisit much later.

A token tax is evidence, not destiny

Suppose two names are equally important in the real world, but the tokenizer learned one from many examples and barely saw the other. The common name might be one token while the rare name becomes five pieces. That difference does not prove unfair behavior by itself, but it is a signal worth inspecting.

cost(common name) = 1 token cost(rare name) = 5 tokens same sentence + rare name = 4 extra tokens of context budget

This is the quiet fairness lesson in tokenization: representation begins before the neural network. A model can only learn from the pieces its tokenizer gives it.

4. Going deeper

Further reading.

Tokenization is small in code and large in consequence. These references move from the core subword algorithms to the libraries and inspection tools used in production systems.

  1. Sennrich, Haddow, and Birch, Neural Machine Translation of Rare Words with Subword Units The influential paper that brought byte pair encoding to neural models.
  2. Kudo, Subword Regularization and Unigram Tokenization The primary source for Unigram language-model tokenization.
  3. Kudo and Richardson, SentencePiece Raw-text subword tokenization and detokenization for neural text processing.
  4. BERT Tokenization Code A readable implementation reference for BasicTokenizer and WordPieceTokenizer.
  5. Hugging Face Tokenizers Course A practical, example-driven tour of training and using modern tokenizers.
  6. tiktoken OpenAI's fast byte-level BPE tokenizer, useful for counting real token costs.
