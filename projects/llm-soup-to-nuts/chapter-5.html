Learning by doing

Transformers: let every token look at every other token.

Chapter 2 ended with a problem. Recurrent models read one token at a time and carry a single note forward, so a clue from far back has to survive a long, lossy journey. The Transformer threw out that journey. Instead of passing a note down a chain, it lets each token reach out and look directly at every other token, all at once.

That single idea, called attention, is the engine inside every modern LLM. The good news: you already have the math. Attention is built from the dot products and softmax you met in Chapter 3. This chapter assembles those pieces into the mechanism, then lets you run a real Transformer in your own browser.

Guiding question: when the model is about to predict the word after the cat sat on the, how does it decide that cat matters more than the? Attention is the answer, and it is just weighted lookup.

1. Attention as weighted lookup

Query, key, value: score the tokens, then blend them.

Attention gives every token three roles, each a short vector. The query is what this token is looking for. The key is what each token offers. The value is the information each token passes along. To update one token, we compare its query against every key with a dot product (the same similarity score from Chapter 3), divide by √d to keep the numbers tame, and run the scores through softmax to get weights that add to 1.

The new representation of the token is then the weighted sum of all the value vectors. High-scoring tokens contribute more. That is the whole mechanism: attention(q) = softmax(q · K / √d) · V. Below, the six tokens of the cat sat on the mat each carry a hand-picked query, key, and value so you can watch the weighted lookup happen. In a real model these vectors are learned.

Where do those vectors come from? The model starts with the token vectors from the embedding table, then learns three projection matrices: W_Q, W_K, and W_V. Multiplying the current token representations by those matrices gives Q = XW_Q, K = XW_K, and V = XW_V. The interactive below skips the learned matrices and gives you the resulting vectors directly, so the attention math itself is visible.

One attention row by hand

Use the default query token, sat. Its query is [0.0, 0.9, 0.1, 0.4]. The key for cat is [0.0, 1.0, 0.0, 0.1]. Their dot product is high because both point strongly at the subject channel.

q_sat dot k_cat = 0.0*0.0 + 0.9*1.0 + 0.1*0.0 + 0.4*0.1 = 0.94 scaled score = 0.94 / sqrt(4) = 0.47 softmax over all six scaled scores gives weights about [14%, 22%, 15%, 15%, 14%, 19%] output_sat = weighted sum of value vectors = [0.38, 0.32, 0.18, 0.31]

2. The attention map

Do it for every token at once, and you get a heatmap.

Run that same weighted lookup with every token taking a turn as the query, and you get a full attention matrix: one row per query token, one column per key token, each cell showing how much that row attends to that column. This is exactly what people mean when they show an "attention pattern." Brighter cells are stronger attention.

Toggle the causal mask from the controls above. Models that generate text left to right are not allowed to peek at the future, so each token may only attend to itself and earlier tokens. That blanks out the upper triangle of the map, and the remaining weights are renormalized. This single rule is what makes a Transformer a next-token predictor rather than a fill-in-the-blank reader.

Mathematically, the mask is simple and a little dramatic: future positions get a score of -Infinity before softmax. Exponentiating -Infinity gives zero, so those future tokens receive exactly no attention weight. Then softmax redistributes the probability mass over the allowed past tokens.

Rows are the query token; columns are the key token. Each number is the attention weight as a percent.

3. Inside a Transformer block

Attention is the heart, but a block has a few more parts.

Attention by itself ignores word order: shuffle the tokens and the dot products are unchanged. So before the first block, every token embedding gets a positional encoding added to it, a signal that says where the token sits in the sequence. Now the cat and cat the look different to the model.

A real model also runs several attention patterns in parallel, called multi-head attention. One head might track the subject of a verb while another tracks nearby punctuation. Their results are joined and passed on. Around each sub-layer sit two workhorses from good neural-network design: a residual connection that adds the input back to the output so signal and gradients flow freely, and layer normalization that keeps the numbers in a stable range. Finally a small feed-forward network processes each position on its own. Stack this block dozens of times and you have a GPT.

One block as equations

Here is a compact post-normalization version that matches the diagram. The shapes matter: if the sequence has T tokens and each vector has d_model numbers, then X is a T by d_model table. Attention mixes information across tokens; the feed-forward network transforms each row independently.

X0 = token_embeddings + positional_embeddings head_i = Attention(X0 W_Qi, X0 W_Ki, X0 W_Vi) multi_head = concat(head_1, ..., head_h) W_O X1 = LayerNorm(X0 + multi_head) X2 = LayerNorm(X1 + FFN(X1))

The residual additions are the skip paths. They let the block change a representation without forcing it to forget what it already knew.

5. Families and variants

One block, many family trees.

The same block powers models that feel quite different, mostly because of how attention is masked and which way the model reads. Decoder-only models like the GPT family use a causal mask and predict the next token; they are the basis of most chat assistants. Encoder-only models like BERT drop the mask, read the whole sentence at once, and are great for understanding tasks. Encoder-decoder models like T5 pair a reader with a writer for translation and summarization.

Beyond the family split, most scaling tricks are about making attention cheaper or the network bigger without paying full price. Multi-query and grouped-query attention share keys and values across heads to save memory. Sliding-window and sparse attention limit how far each token looks. A mixture of experts swaps the single feed-forward network for many, using only a few per token. The core you just built does not change; these are ways to afford more of it.
