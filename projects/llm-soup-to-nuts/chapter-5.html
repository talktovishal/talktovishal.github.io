Learning by doing

Transformers: how ChatGPT looks back at words :-).

The Transformer from first principles: let's build the attention machinery that lets an LLM use context.

I type a question into ChatGPT, and a few seconds later it writes back like it has kept track of the whole conversation. How does that happen? How does the word it know what it refers to? How does the model remember something I wrote many words ago? Let's build the core idea ourselves.

Here is the picture I want you to keep in your head. Older models often felt like a line of people whispering a message forward: each word passes a small note to the next word, and we hope the important clue survives until the end. The Transformer changes the game. It lets each token look back directly at the other tokens and ask, "which of you matters for me right now?" That move is called attention. It sounds fancy, but by the end of this chapter it will feel like a very careful weighted lookup.

This chapter is built around the "Looking Inside Large Language Models" chapter of Hands-On Large Language Models. I am using it as the route map, but not copying it. The goal here is the same style as Chapter 1: rebuild the idea slowly, with our own little examples, until the words query, key, value, and KV cache actually mean something.

Guiding question to keep in your pocket: when the model reads Sarah fed the cat because it and has to figure out what it means, how does it decide that it points at the cat and not at Sarah? Attention is the answer. Woohoo, this is the mechanism we are going to take apart :-)

1. One token at a time

Print one token, attach it, and ask again.

Let me clear up the first big misconception. When an LLM writes a paragraph, it does not print the whole paragraph in one shot. It predicts one token, attaches that token to the end of the prompt, and then asks the same question again: "given everything so far, what token should come next?" Predict, append, repeat. That is the whole loop.

Each trip through the neural network has a name: a forward pass. That just means the input flows through the model, the model does its calculations, and one next-token prediction comes out. So when we say "the model wrote an email," what really happened is many forward passes, each one adding one token to the growing text.

There is a word for a model that feeds its own output back in as input to make the next prediction: it is called autoregressive. (That is the same word that explains why a chatbot streams its answer to you piece by piece instead of all at once — it is genuinely making the next piece only after it has committed to the last.) 2. The forward pass

Text goes in; next-token scores come out.

So what actually happens inside one forward pass? I like to split the model into three big pieces. First comes the tokenizer (your friend from Chapter 3). It breaks the text into tokens and hands each one an integer ID. Each ID then looks up a row of numbers — its embedding — in a big table the model learned during training.

Those embedding vectors flow through the second part: a tall stack of Transformer blocks, often dozens of them, each one refining the representation a little more. Think of this stack as the model's working area. This is where it mixes context, applies learned patterns, and slowly turns plain token vectors into useful next-token evidence.

At the very top sits the third part, the language modeling head (or "LM head"). It is a single layer that turns the stack's output into one number for every token in the vocabulary — tens of thousands of them — and softmax turns those numbers into a probability for each possible next token. That probability bar chart is the model's answer. It is the very same kind of picture you built back in Chapter 1.

3. Choosing the next token

The model gives chances; we still have to choose.

The LM head hands us a probability for every token, but we still have to pick one token to actually print. This is the same question from Chapter 1: after we have all the bars, do we always choose the tallest bar, or do we let the smaller bars have a chance too? The rule we use is called the decoding strategy.

The simplest rule is greedy decoding: always take the single highest-probability token. It sounds ideal, but it tends to produce flat, repetitive text. A livelier approach is to sample — roll a weighted die so that a token with 40% probability gets picked about 40% of the time, while less likely tokens still get an occasional turn. A knob called temperature controls how adventurous the sampling is: low temperature hugs the top choice, high temperature spreads the love around. (Greedy is just temperature turned all the way down; we save the full temperature story for Chapter 6.)

4. Parallel streams and context

Every token gets its own little track.

Here is one reason Transformers became such a big deal. The old line-of-people picture processes words one after another. A Transformer gives each token its own little stream of computation, and many streams can run at the same time. Modern GPUs love that. It is like opening many checkout lanes instead of making every customer wait in one line.

The number of streams a model can run at once is its context length. A model with a 4,000-token context has room for 4,000 streams; ask for more and it simply has nowhere to put them. This is why "context window" is a number you hear quoted for every model. It is the size of the reading space the model has available at once.

One subtlety worth pausing on: to predict the next token, the model only uses the output of the last stream. So why bother computing all the others? Because attention in that last stream reaches back and reads the earlier streams. Their work is not wasted. It is the evidence the final token leans on.

Exercise: count the streams

A prompt has 6 tokens and the model's context length is 10 tokens. How many token streams run during the forward pass, and which stream feeds the LM head for the next-token prediction?

Answer: 6 streams run, one per input token. The last stream is the one whose final vector feeds the LM head, but attention lets that last stream pull useful information from the earlier five.

5. The KV cache

Don't do the same homework again and again.

Remember the loop from the first hands-on demo above: to generate the next token, we append the last one and run another forward pass. But wait a second. The first hundred tokens did not change! Recomputing all their attention ingredients from scratch on every step is like doing yesterday's homework again before starting today's.

The fix is the KV cache. Inside attention, every past token produces a key and a value vector (you will build these by hand two sections from now). Those never change once a token is fixed, so we simply save them. On the next step the model computes the key and value for only the one new token and reuses everything cached for the past. The heavy work per step drops from "redo the whole prefix" to "do one new token."

The payoff is not subtle. In the book this chapter follows, generating 100 tokens took about 4.5 seconds with the cache on and roughly 21.8 seconds with it off. Slide the length below and watch the total amount of work explode without the cache and stay gentle with it on. This is one of those engineering tricks that sounds boring until you feel the speed difference.

6. Inside the block

Each block has two jobs: remember patterns and gather context.

Time to open one of those stacked blocks. This is where the model spends most of its thinking time. A Transformer block has two important pieces: a self-attention layer and a feed-forward neural network. They sound similar, but I want you to think of them as doing two different jobs.

The feed-forward network is the model's memory and its reasoning-by-pattern. Feed a model The Shawshank and it will happily continue with Redemption; that stored association lives in the feed-forward layers, learned from seeing the phrase countless times. But it is not a lookup table — the same machinery lets the model interpolate, handling inputs it never saw exactly. Recall plus generalization. Useful, but not enough.

The attention layer is the context-gatherer. Memory alone is not enough, because meaning depends on the rest of the sentence. Take The dog chased the squirrel because it was scared. To continue after it, the model has to know whether it is the dog or the squirrel. Attention is the mechanism that reaches back, weighs the earlier words, and folds the right one into it's representation. That is the job we will build by hand next. This is the good part.

Quick check: in … the squirrel because it, which part of the block is responsible for working out that it means the squirrel?

Exercise: sort the jobs

Answer: resolving it and blending from an earlier noun are attention jobs because they mix positions. Remembering the movie title and applying a learned per-token pattern are feed-forward jobs.

7. Query, key, value

Ask a question, find the matching keys, and blend the answers.

Now the heart of the whole thing. Suppose the current token is it. It is basically asking, "what thing in the sentence am I standing in for?" Attention turns that question into three little vectors for every token. In a real model these vectors are learned; in our tiny model I set them by hand so we can see the machinery clearly.

  • Query — the question the current token is asking.
  • Key — the label each token holds up: "here is what I can help with."
  • Value — the information that token will hand over if it is useful.

With those in hand, attention runs in exactly two steps. Step one, relevance scoring: take the current token's query and compare it against every token's key with a dot product. A big dot product means "these two match." We divide by √d to keep the numbers tame, then run the scores through softmax so they become weights that add up to 1. Step two, combining information: multiply each token's value by its weight and add them all up. Tokens that scored high contribute most. The sum is the token's new, context-aware representation. In one line: attention(q) = softmax(q · K / √d) · V.

Below is our running sentence, Sarah fed the cat because it, with hand-picked query/key/value vectors so the mechanism is visible (in a real model these are learned). The default query is it. Watch where it looks. My bet is that you will see the idea click as soon as the cat bar jumps ahead.

Why "it" finds "cat", by hand

The query for it leans hard on the "animate / living-thing" channel, because a pronoun like it is hunting for a thing it could stand in for. The key for cat leans on that same channel. So their dot product is large. The key for fed is more verb-like, so it does not match the question very well.

q_it · k_cat = high → "it" and "cat" match strongly q_it · k_the = medium → a determiner, mildly relevant q_it · k_fed = ~0 → a verb, nothing to offer "it" softmax of all six scores → cat wins most of the weight output_it = weighted sum of values ≈ mostly cat's value

8. The attention map

Let every token take a turn asking the question.

Run that same scoring with every token taking a turn as the query, and you get a full attention matrix: one row per query token, one column per key token, each cell showing how much that row attends to that column. This grid is exactly what people mean when they show an "attention pattern." Brighter cells are stronger attention. It is the same calculation you just did for it, repeated for every token.

Now toggle the causal mask. A model that writes left to right is not allowed to peek at the future — when it is processing a token, the tokens that come after it do not exist yet. So each token may only attend to itself and the tokens before it. That blanks out the upper triangle of the map and the remaining weights are renormalized. This one rule is what makes a model a next-token predictor (like GPT) rather than a fill-in-the-blank reader (like BERT, whose B is for "bidirectional" precisely because it looks both ways).

The trick under the hood is tidy: future positions get a score of -∞ before softmax. Since e-∞ is zero, those cells receive exactly no weight, and softmax shares the probability out over the allowed past. In plain English: cover the future before the model can cheat.

9. Position information

The model also needs to know word order.

We have one more quiet problem to solve. Attention compares vectors, but by itself it does not know that dog bites man and man bites dog mean different things. Same tokens, totally different story! A language model needs some way to know where each token sits, or every sentence starts to look like a bag of words.

Early Transformers added a positional embedding near the bottom of the model: token vector plus "I am position 1," "I am position 2," and so on. Modern decoder LLMs often use rotary positional embeddings, or RoPE, instead. I like to think of RoPE as putting a tiny position-dependent twist into the query and key vectors before they are compared. The dot product then asks two questions at once: do these tokens match, and are they in the right relative place?

That timing matters. RoPE lives inside attention, right before relevance scoring, so the model can ask questions like "does this key match my query, and is it the right distance away?" It also plays more nicely with long contexts and packed training examples, where absolute position numbers alone can be a clumsy signal.

Quick check: why is position information necessary if attention already sees all the tokens?

Exercise: spot the order problem

Compare the cat chased the dog with the dog chased the cat. If a model kept only token identities and threw away positions, what crucial fact would it lose?

Answer: it would lose who did the chasing and who got chased. Attention needs position information so the same tokens in a different order produce different representations.

10. Multi-head attention

Use several attention heads, because one question is not enough.

One attention pattern is not enough. A sentence has many kinds of relationship at once — a pronoun and its referent, a verb and its subject, a word and the punctuation nearby. So a Transformer runs attention several times in parallel, and each copy is called an attention head. Each head gets its own learned query/key/value projections, so each one learns to care about a different kind of pattern.

Mechanically it is "split, attend, combine." The token's vector is split across the heads, each head does the two-step attention you just built, and their results are stitched back together. It is like asking several friends to read the same sentence, each with a different highlighter. One marks pronouns, one marks previous words, one marks subjects, and together they give you a richer picture.

Pick a query token below and compare three heads we hand-designed. One chases meaning (the coreference head that sends it to cat), one is a "look at the token just before me" head, and one always anchors on the sentence's subject. Same words, three different stories.

12. Recent improvements

The original idea still stands, but engineers keep making it faster.

The Transformer you just built is, remarkably, still the core of a 2024-era model like Llama 3. But a few well-earned tweaks make modern models faster and cheaper. Most of the tweaks answer one of two practical questions: can we make attention less expensive, and can we make the block easier to train? Here is a field guide, aka the names you will keep seeing in LLM architecture diagrams.

Local / sparse attention Full attention is the costliest step, since every token looks at every other. Sparse and sliding-window attention let a token see only a nearby slice. GPT-3 interleaved full and sparse blocks to keep quality while cutting cost.
Multi-query & grouped-query attention Instead of every head owning a full set of keys and values, heads share them — all of them (multi-query) or in groups (grouped-query). Far less memory to move at inference. Llama 2 and 3 use grouped-query.
Flash Attention Same math, smarter bookkeeping: it is careful about what gets moved between a GPU's fast and slow memory, giving big speedups for both training and inference without changing the result.
Pre-norm, RMSNorm, SwiGLU Modern blocks normalize before each sub-layer instead of after, use the simpler, faster RMSNorm, and swap the old ReLU for a SwiGLU activation in the feed-forward network.
Document packing Training contexts are long but many documents are short, so several are packed into one context (with separators) to avoid wasting the window on padding — a small efficiency trick with a real impact at scale.

Quick check: what is the main thing grouped-query attention saves compared with classic multi-head attention?

Exercise: pick the right upgrade

Match each bottleneck to a likely improvement: "GPU memory traffic dominates attention," "the model needs cheaper inference-time key/value storage," "many training examples are shorter than the context window," and "very long documents make full attention expensive."

Answer: FlashAttention targets GPU memory traffic; grouped-query or multi-query attention shrinks key/value storage; document packing reduces wasted context during training; local or sparse attention trims the full attention pattern for long sequences.
