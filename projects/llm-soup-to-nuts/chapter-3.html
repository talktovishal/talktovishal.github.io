Learning by doing

Tokenization: how text becomes pieces a model can read :-).

I type unbelievable, and it feels like one word to me. To the model, it might be one piece, three pieces, or a handful of raw bytes. That hidden choice changes cost, speed, context length, and even some of the strange mistakes people blame on LLMs. So let's stop treating tokenization like boring setup and build the pieces ourselves.

Let me make the hidden step plain: a neural model never sees your letters. Not one. It sees token IDs - plain integers that index into its embedding table - and nothing else. Before any math can happen, a small separate program called the tokenizer chops your text into known pieces and looks up their numbers. Pick the pieces badly and a common word balloons into three tokens; pick them well and a word the model has never seen still spells itself out of familiar parts.

Guiding question: if the word unbelievable never appeared in training, how can the model still handle it? Keep that question in the back of your mind. The answer is not that the model owns a giant dictionary. It is that the word was stored as parts the model has seen many times.

Before we dive in, a few needed prerequisites are in order. Tokenization sits before all the neural-network math, so this chapter is lighter on calculus and heavier on plain bookkeeping. A little comfort with the following will help - the links are quick refreshers if anything feels new to you.

  • Chapter 1. We reuse the idea of a vocabulary and the fact that every model reads token IDs (integers), never letters.
  • Chapter 2. Each token ID looks up a row in the embedding table - that lookup is the very first thing the model does with a token.
  • Counting and bytes. The one genuinely new idea is that text is ultimately a list of numbers from 0 to 255. If you can count how often a pair shows up, you can follow every algorithm here.

What you do not need: any of the gradient or softmax math - that arrives right after this, in Chapter 4. A tokenizer does not learn by backprop - it just counts, merges, and looks things up.

1. Why tokenization matters

The same sentence can be cheap or expensive, depending on the pieces.

Chapter 1 used the friendliest possible tokenizer: split the text on spaces, then count the words. That was exactly right for our first model, but it hid a real decision. Should the unit be a whole word, a single character, a raw byte, or a subword chunk in between? Each answer changes how many tokens the same sentence becomes, and token count is what you pay for in money, memory, and latency.

The picture I keep in my head is cutting the same sentence into different sized building blocks. Whole words are easy to read, but rare words have no entry and break. Characters and bytes can spell anything, but they make long sequences that are slow and expensive. Subwords are the middle path used by almost every modern model: keep frequent pieces whole, split rarer words into reusable parts. Type into the box and watch the four views disagree on the very same text.

There is a useful little budget hiding here, and it is worth feeling once before we move on. If a prompt has T tokens, a Transformer attention layer compares tokens to tokens, so the attention score table has about T * T cells. Doubling the token count does not merely double that table; it makes it roughly four times larger. The tokenizer also chooses the size of the embedding table: with V vocabulary pieces and d numbers per vector, the first table has V * d learned numbers. Tokenization is therefore a tradeoff: a larger vocabulary can shorten sequences, but it also creates more rows the model has to learn.

Token cost by hand

Imagine one sentence becomes 12 tokens with a good subword tokenizer, but 20 tokens with a byte-heavy tokenizer. The text looks the same to you, but the Transformer does more pairwise comparisons.

attention cells with 12 tokens = 12 * 12 = 144 attention cells with 20 tokens = 20 * 20 = 400 400 / 144 = 2.8x as many token-to-token scores

This is why a word that splits into many pieces has a real token tax: it can cost more memory, more time, and more money even before the model starts reasoning about meaning.

2. Down to the bytes

Before tokens, text falls all the way down to bytes.

The opening section let four rules - word, character, byte, subword - argue over the same sentence. But it skipped a fair question: what is a "piece" made of? When I first learned this, it felt almost too low-level, like we had left language behind. But this is exactly where Karpathy starts his tokenizer walkthrough, and it is the right place to start. To a computer, a string is a sequence of Unicode code points. Unicode is a giant catalogue - about 150,000 characters across 161 scripts, and still growing every single year - that hands one integer to every character humanity writes, from A to あ to 👍. The letter H is code point 72; that thumbs-up is 128,077.

So why not feed the model those code-point integers directly and call it a day? Two reasons, and both bite. First, the vocabulary would be enormous - 150,000 rows in the embedding table before the model has learned anything. Second, and worse, Unicode is a living standard: new characters land every year, so today's numbering is a moving target. We want a base that is smaller, fixed, and universal.

That base is UTF-8, the encoding nearly all of the internet already speaks. UTF-8 takes each code point and writes it out as 1 to 4 bytes - and a byte is simply a number from 0 to 255. Plain English letters take 1 byte each (UTF-8 is backward-compatible with old ASCII); accented letters and most scripts take 2 or 3; an emoji takes 4. So here is the bedrock fact for the whole chapter: underneath every tokenizer, any text at all - English, Korean, source code, emoji - is just a stream of bytes, each between 0 and 255. A byte-level base needs a vocabulary of exactly 256, it can spell literally anything ever written, and nothing can crash it with a character it has never seen.

There is one catch, and it is the entire reason the next section exists. If we stop at raw bytes, sequences get long - that emoji alone is four tokens, an everyday English word is several - and a Transformer can only attend to so many tokens at once. Raw bytes squander that budget. So we keep the universal 256-byte foundation and build on top of it: glue the byte pairs that show up most often into new, longer tokens. That gluing algorithm is byte-pair encoding, and now it should feel less like a trick and more like the obvious next move.

One string, all the way down to bytes

Take the four-character string Hi 👍 - an H, an i, a space, and a thumbs-up. Python's ord() gives each character its Unicode code point; UTF-8 then rewrites each code point as bytes:

code points: H = 72 i = 105 space = 32 👍 = 128077 UTF-8 bytes: [72] [105] [32] [240, 159, 145, 141] 4 characters → 7 bytes, because the emoji alone needs 4 bytes

This is not trivia. The model never sees an "emoji" - it sees the four bytes 240 159 145 141. Almost every quirk later in this chapter - shaky spelling, broken arithmetic, expensive Korean - begins right here, with how these bytes get grouped into tokens.

3. Tokenizer algorithms

Let's train the tokenizer's pieces from real text.

First, here is a framing that took me a while to appreciate: the tokenizer is a completely separate object from the language model. It has its own training set of documents, it is trained exactly once as a pre-processing step, and then it is frozen. Only afterwards does the giant model train on the tokens it produces. The model does not get to renegotiate the pieces later. So everything in this section happens before the neural network even exists - the tokenizer just learns, from raw bytes, which pieces are worth a slot in the vocabulary. (A nice side effect: whatever mix of languages and code you feed the tokenizer decides which pieces get merged, and therefore how short each language's sequences end up.)

The workhorse algorithm is byte-pair encoding (BPE), and it is almost insultingly simple - I mean that in the best way. Start from the 256 byte tokens. Count every adjacent pair in the corpus, find the single most common one, and mint a brand-new token for it - id 256, then 257, then 258 - that stands for that pair. Replace every occurrence, then count again. Each merge makes the sequence a little shorter and the vocabulary a little bigger, and you stop when the vocabulary reaches a size you chose (GPT-4's is about 100,000). Once trained, the tokenizer can encode (text → token IDs) and decode (token IDs → text) - the two directions it forever translates between.

One twist that real tokenizers add is worth knowing, because it is one of those boring-looking details that quietly changes everything. Run BPE completely naively and you get junk merges: dog, dog., dog!, and dog? each become their own near-duplicate token, fusing a word onto its punctuation. GPT-2's fix is to first split the text with a regular expression - keeping letters, numbers, punctuation, and whitespace in separate chunks - and to only ever merge within a chunk. That single rule is why a comma never welds onto the word before it, and why GPT-4 later tuned the same pattern (capping number-merges at three digits, grouping runs of whitespace) to tokenize code far more efficiently than GPT-2 did.

This lab is small, but real. BPE repeatedly merges the most frequent neighboring pair. WordPiece-style encoding greedily takes the longest known piece. Unigram-style encoding picks the most likely segmentation from a learned set of pieces. Byte-level encoding skips training and uses the raw UTF-8 bytes from the last section. Continuation pieces are shown with a ## prefix, and the merges train on the corpus you picked above, so they are genuine.

Two tokenizer algorithms by hand

BPE is a counting algorithm. Start with characters, count adjacent pairs, merge the most frequent pair, then count again. On the tiny corpus hug hug hug hugs hugging hugging, the pairs h u and u g both appear 6 times, so the first merge can turn h + u into hu.

score_BPE(pair) = count(pair in the training corpus) first merge: h + u -> hu because count(h u) = 6 next useful merge: hu + g -> hug, so hug becomes one reusable piece

Unigram thinks differently. It keeps many possible pieces and asks which segmentation has the best total probability. In practice it uses dynamic programming; by hand, you can read it as "choose the pieces with the smaller surprise."

score(segmentation) = sum(-ln p(piece)) un + believ + able: 3.22 + 4.61 + 3.00 = 10.83 nats unbelievable as one rare piece: -ln(0.000001) = 13.82 nats

Lower score wins, so this toy Unigram model prefers the familiar parts. BPE asks "what pair was frequent?"; Unigram asks "what full spelling is most probable?"

4. Why models break: the tokenization quirks

Many "the model is dumb" moments are really tokenizer moments.

Here is the part Karpathy saves for the end of his talk, and it is the most fun: a startling number of an LLM's famous failures are not failures of the neural network at all. They are failures of tokenization, hiding in plain sight. Once I started checking the tokens, the weirdness stopped being mysterious and starts being almost obvious. Remember the bedrock fact - the model never sees your letters, it sees chunks - so anything that depends on the letters inside a chunk is hard for it, and anything that bloats the number of chunks is expensive for it.

Pick a case in the lab below and watch the token stream; sometimes the visible text and the tokenized input are far less alike than they look. But first, let's make a little field guide. Each quirk below is a real, documented behavior, and every one traces straight back to how the bytes got grouped.

A token tax is evidence, not destiny

Suppose two names are equally important in the real world, but the tokenizer learned one from many examples and barely saw the other. The common name might be one token while the rare name becomes five pieces. That difference does not prove unfair behavior by itself, but it is a signal worth inspecting.

cost(common name) = 1 token cost(rare name) = 5 tokens same sentence + rare name = 4 extra tokens of context budget

This is the quiet fairness lesson in tokenization: representation begins before the neural network. A model can only learn from the pieces its tokenizer gives it.

Can't spell or reverse words A word like DefaultStyle is one fat token, so its letters are hidden. Ask how many L's it has, or to reverse it, and the model fumbles.
Shaky arithmetic Numbers chunk arbitrarily - 127 is one token, 677 is two - so digit-by-digit math has no steady digits to grab. Llama deliberately splits every digit.
Non-English is expensive "hello how are you" is 5 tokens; its Korean translation is 15. Three times the tokens burns three times the context and gives less signal per word.
Python broke GPT-2 Every indentation space was its own token, bloating code until it ran out of context. GPT-4 fixed it by grouping runs of whitespace.
Trailing whitespace trips it A space is normally glued to the front of the next word ( of), so a lone trailing space is out of distribution and worsens the completion.
Special tokens are a trap Strings like <|endoftext|> are single special tokens; slipping one into a prompt can halt or derail the model - a genuine attack surface.
SolidGoldMagikarp A Reddit username that earned its own token in the tokenizer's data but never appeared in the model's, so its embedding row stayed random. Mention it and the model babbles.
YAML beats JSON The same data costs fewer tokens in YAML than in JSON, so in a pay-per-token world YAML is literally cheaper to send.

5. Going deeper

Further reading.

Tokenization is small in code and large in consequence. If you read only one thing after this, make it Karpathy's tokenizer talk - this whole chapter is my attempt to walk you through that same ladder slowly enough that you can own it. The rest of these references move from the core subword algorithms to the libraries and inspection tools used in production systems.

  1. Karpathy, Let's build the GPT Tokenizer (video) The two-hour talk this chapter is built around - bytes, BPE, the GPT-2/GPT-4 regex, and every quirk, built up from scratch.
  2. Sennrich, Haddow, and Birch, Neural Machine Translation of Rare Words with Subword Units The influential paper that brought byte pair encoding to neural models.
  3. Kudo, Subword Regularization and Unigram Tokenization The primary source for Unigram language-model tokenization.
  4. Kudo and Richardson, SentencePiece Raw-text subword tokenization and detokenization for neural text processing.
  5. BERT Tokenization Code A readable implementation reference for BasicTokenizer and WordPieceTokenizer.
  6. Hugging Face Tokenizers Course A practical, example-driven tour of training and using modern tokenizers.
  7. tiktoken OpenAI's fast byte-level BPE tokenizer, useful for counting real token costs.
  8. minbpe Karpathy's minimal, readable BPE implementation - the code companion to the video above.
  9. Tiktokenizer Paste any text and watch how different models slice it into tokens, live in the browser.
