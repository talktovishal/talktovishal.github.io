Learning by doing

Tokenization: How text is converted into tokens that a model can read.

I type unbelievable, and it feels like one word to me. To the model, it might be one piece, three pieces, or a handful of raw bytes. That choice influences cost, speed, context length, and even some of the strange mistakes people blame on LLMs. Hence, it's important to understand how tokenization works. In this chapter, we'll understand the details of tokenization by building a simple tokenizer from scratch and exploring all its nuances.

A neural model never sees your letters directly. Not one. It sees token IDs - plain integers that map into its embedding table - and nothing else. Before any LLM math can happen, a separate program called the tokenizer chops your text into known pieces and looks up their numbers. Each number represents a specific subword or token that the model has learned to recognize. For example, the word "unbelievable" might be tokenized into three tokens: "un", "believ", and "able". Remember, in Chapter 2, we learnt how a particular word get's mapped to a list of numbers aka vector. We now learn this process in detail and see modern LLMs, first break down the text into subword pieces and then use their vectors during their processing.

Guiding question: if the word supercalifragilisticexpialidocious :-) never appeared in training, how can the model still handle it? The answer is not that the model owns a giant dictionary of all possible words. It is that the word was stored as smaller parts, each of which the model has seen many times.

  • Chapter 1. We reuse the idea of a vocabulary and the fact that every model reads token IDs (integers), never letters.
  • Chapter 2. Each token ID looks up a row in the embedding table - that lookup is the very first thing the model does with a token.
  • Counting and bytes. The one genuinely new idea is that text is ultimately a list of numbers from 0 to 255. If you can count how often a pair shows up, you can follow every algorithm here.

What you do not need: any of the gradient or softmax math - that arrives right after this, in Chapter 4. A tokenizer does not learn by backprop - it just counts, merges, and looks things up.

1. Why tokenization matters

The same sentence can be cheap or expensive, depending on the pieces.

Chapter 1 used the friendliest possible tokenizer: split the text on spaces, then count the words. That was exactly right for our first model, but it hid a real decision. Should the unit be a whole word, a single character, a raw byte, or a subword chunk in between? Each answer changes how many tokens the same sentence becomes, and token count is what you pay for in money, memory, and latency.

The picture I keep in my head is cutting the same sentence into different sized building blocks. Whole words are easy to read, but rare words have no entry and break. Characters and bytes can spell anything, but they make long sequences that are slow and expensive. Subwords are the middle path used by almost every modern model: keep frequent pieces whole, split rarer words into reusable parts. Type into the box and watch the four views disagree on the very same text.

I am going to break my rule and talk about Transformers - the main architecture used in modern LLMs. It's important to understand the impact of tokenization on the performance of these models. For now, the important thing to remember is during processing, each token is influenced by its context and the tokens around it. It does that by mutiplying the token's embedding with the embeddings of the surrounding tokens. Hence, if a sentence has T tokens, a Transformer attention layer compares tokens to tokens, so the attention score table has about T * T cells. Doubling the token count does not merely double that table; it makes it roughly four times larger. The tokenizer also chooses the size of the embedding table: with V vocabulary pieces and d numbers per vector, the first table has V * d learned numbers. Tokenization is therefore a tradeoff: a larger vocabulary can shorten sequences, but it also creates more rows the model has to learn.

Token cost by hand

Imagine one sentence becomes 12 tokens with a good subword tokenizer, but 20 tokens with a byte-heavy tokenizer. The text looks the same to you, but the Transformer does more pairwise comparisons.

attention cells with 12 tokens = 12 * 12 = 144 attention cells with 20 tokens = 20 * 20 = 400 400 / 144 = 2.8x as many token-to-token scores

This is why a word that splits into many pieces has a real token tax: it can cost more memory, more time, and more money even before the model starts reasoning about meaning.

2. Down to the bytes

The most simplest tokenization strategy - breaking text all the way down to bytes.

The opening section let four rules - word, character, byte, subword - argue over the same sentence. But it skipped a fair question: what is a "piece" made of? When I first learned this, it felt almost too low-level, like we had left language behind. Hence, it is the right place to start. To a computer, a string is a sequence of Unicode code points. Unicode is a giant catalogue - about 150,000+ characters across 161 scripts, and still growing every single year - that hands one integer to every character humanity writes, from A to あ to 👍. The letter H is code point 72; that thumbs-up is 128,077.

So why not feed the model those code-point integers directly and call it a day? Two reasons, and both bite. First, the vocabulary would be enormous - 150,000 rows in the embedding table before the model has learned anything. Second, and worse, Unicode is a living standard: new characters land every year, so today's numbering is a moving target. We want a base that is smaller, fixed, and universal.

That base is UTF-8, the encoding nearly all of the internet already speaks. UTF-8 takes each code point and writes it out as 1 to 4 bytes - and a byte is simply a number from 0 to 255. Plain English letters take 1 byte each (UTF-8 is backward-compatible with old ASCII); accented letters and most scripts take 2 or 3; an emoji takes 4. So here is the bedrock fact for the whole chapter: underneath the text itself, any text at all - English, Korean, source code, emoji - is just a stream of bytes, each between 0 and 255. A byte-level base needs a vocabulary of exactly 256, it can spell literally anything ever written, and nothing can crash it with a character it has never seen.

There is one catch, and it is the entire reason the next section exists. If we stop at raw bytes, sequences get long - that emoji alone is four tokens, an everyday English word is several - and a Transformer can only attend to so many tokens at once. Raw bytes squander that budget. So GPT-style byte-level tokenizers keep the universal 256-byte foundation and build on top of it: glue the byte pairs that show up most often into new, longer tokens. That gluing algorithm is called byte-pair encoding (BPE).

One string, all the way down to bytes

Take the four-character string Hi 👍 - an H, an i, a space, and a thumbs-up. Python's ord() gives each character its Unicode code point; UTF-8 then rewrites each code point as bytes:

code points: H = 72 i = 105 space = 32 👍 = 128077 UTF-8 bytes: [72] [105] [32] [240, 159, 145, 141] 4 characters → 7 bytes, because the emoji alone needs 4 bytes

This is important. The model never sees an "emoji" - it sees the four bytes 240 159 145 141. Almost every quirk later in this chapter - shaky spelling, broken arithmetic, expensive Korean - begins right here, with how these bytes get grouped into tokens.

3. Tokenizer algorithms

Let's dive in and become the expert :-).

Let's slow down to deeply understand the concepts. If this part feels fuzzy, and if it stays fuzzy a lot, then the understanding of how an LLM actually works will stay fuzzy too. To reiterate: a tokenizer is a separate object from the language model. It has its own training corpus, learns a vocabulary once, assigns integer IDs to those pieces, and then freezes. Only after that does the giant neural model train on the token IDs the tokenizer produces. The model does not get to renegotiate the pieces later.

That makes the tokenizer's choices matter: they decide how many tokens a sentence costs, how much of your context window it eats, and whether a rare word survives whole or shatters into fragments. Its job is also a budget decision: which chunks of text deserve their own row in the vocabulary? Make the chunks too big and the vocabulary explodes. Make them too small and every sentence becomes a long, expensive chain. Subword tokenization was invented to live in the middle.

First, "word" is not a clean unit.

Is pool, one word, or a word plus a comma? Is I'm one word, or I plus 'm? Chinese, Japanese, and Thai do not always mark word boundaries with spaces, and words like longer contain meaningful pieces such as long and -er. So a tokenizer is never merely "split on spaces"; it is choosing a boundary policy before the model sees anything.

Possible unit Why it is tempting What breaks How subwords use the good part
Whole words A 20-word sentence is about 20 tokens, which is short and readable. The vocabulary can climb toward 500k-1M entries. The embedding table has V * d learned numbers, and the output softmax must score all V choices. Rare words, typos, names, and new compounds fall into [UNK] unless they were memorized. They all get the same generic representation. Keep the efficiency of common whole words when they earn a vocabulary slot, but split rare words into familiar pieces.
Characters or bytes The vocabulary is tiny: roughly 100 common characters, or exactly 256 UTF-8 byte values. Sequence length explodes. A 20-word sentence can become 120+ character tokens, which burns context window and makes attention's T * T table much larger. The model also has to rebuild meaning from tiny fragments. Keep the coverage idea: any text can still be spelled from small pieces, while common chunks collapse into shorter tokens.
Subwords They are the compromise: common chunks like do, n't, micro, or phobia get their own IDs. The tokenizer has to learn which chunks are worth the limited vocabulary slots, and different algorithms make different choices. This is the compromise itself: a manageable vocabulary, which falls back to shorter sequences when needed, and no panic when a new word appears.

To summarize, whole-word vocabularies also keep growing as you read more text. New names, products, places, typos, and technical terms keep appearing; this steady growth is the intuition behind Heaps' law (also called Herdan's law). Subwords are attractive because they let the model reuse smaller pieces instead of reserving a brand-new row for every new surface form.

A useful summary number is the fertility rate: average tokens per word. Around 1.5-3 tokens per word is a healthy middle zone for many single-language settings. If the rate is too high, the model is paying a token tax. If it is too low because the vocabulary is enormous, the model pays in parameters and output scoring instead. Multilingual models often need larger vocabularies because many scripts and word-building systems have to share the same budget.

Separate the boundary policy from the learning rule.

Here is the distinction that trips up almost everyone, so let's nail it down before going further: BPE, WordPiece, and SentencePiece are not three perfectly parallel things. BPE and WordPiece are vocabulary learning rules. SentencePiece is a raw-text tokenizer system that can train either a BPE model or a Unigram language-model tokenizer. This is why papers and blog posts can sound slippery: sometimes "SentencePiece" means the toolkit, and sometimes people use it as shorthand for a SentencePiece Unigram model. So the better question is not "BPE or SentencePiece?" It is: what does the tokenizer do before it learns pieces, and what rule does it use to learn them? That first job, how the raw text is split and cleaned up before any pieces are learned, is called the boundary policy (you will also see it called pre-tokenization). The merge-or-score step that follows is the separate learning rule.

The Tokenizer System = Boundary Policy + Base Alphabet + Learning Rule + Runtime Encoder
GPT-style tokenizer system = regex chunks + UTF-8 bytes + BPE merges + replay merge list WordPiece system = word + punctuation split + characters + association-score merges + longest-match SentencePiece BPE system = raw text with visible spaces + BPE merges + replay merge list SentencePiece Unigram system = raw text with visible spaces + scored pieces + Viterbi route

This is the place where WordPiece and SentencePiece part ways the most, and it is worth slowing down for. WordPiece (the BERT-style system) depends on a pre-tokenizer: a separate first pass that splits the text into whole words by looking for spaces and punctuation, and only then breaks each word into subwords (un + ##friend + ##ly). The catch is the hidden assumption that spaces mark where words end. Languages like Chinese, Japanese, and Thai do not separate words with spaces, so this path needs custom, language-specific rules just to prepare the text, and because it discards spaces to find the word edges, you cannot always rebuild the exact original string from the tokens. Hence, you need language-specific word breakers for these languages. That's a significant investment in maintaining and supporting these systems.

SentencePiece removes that first pass entirely. It feeds the raw Unicode stream straight into the learning rule (BPE or Unigram) and treats the space itself as an ordinary character, escaping it as the visible marker ▁ (U+2581). With nothing relying on spaces to find word boundaries, the very same logic handles English, Chinese, Python, and math notation with zero language-specific rules (it is language-agnostic), and because the space survives as a real token, the pieces merge back into the exact original text, so decoding is lossless and reversible. Bypassing language-specific pre-processing and making spaces ordinary characters is the breakthrough that made SentencePiece the default boundary policy for modern multilingual LLMs.

Let's see how these systems come from. Byte-pair encoding began back in 1994 as a data-compression trick: repeatedly fuse the most common adjacent byte pair into a new symbol. Two decades later, Sennrich and colleagues reused that merge idea for machine translation, where a fixed vocabulary keeps meeting rare names and compounds. Splitting them into reusable subword pieces beats dropping them in an unknown bucket. Google's GNMT answered the same need with WordPiece, which favors pieces that are distinctively paired rather than merely frequent, though it, like BERT after it, still leaned on a language-specific pre-tokenizer to split the words first. In 2018 SentencePiece dropped that assumption entirely, training straight from the raw character stream (spaces kept as the visible marker ▁) so space-free languages and mixed scripts can share one reversible tokenizer with no custom pre-processing.

Now lets see why LLMs like GPT-2 start to leverage BPE. An open-ended web model has to give some probability to any Unicode string, so it used byte-level BPE: bytes guarantee coverage, BPE adds compression. BERT's WordPiece fit a different job (distinctive subwords for masked-token understanding), and SentencePiece fit yet another (raw multilingual text with reversible spaces).

This split is still how it works today. DeepSeek-V3 and Qwen2.5 both use byte-level BPE with large (128K–150K) vocabularies, while Baichuan 2 runs BPE inside SentencePiece with UTF-8 byte fallback. One more reminder that "SentencePiece BPE" is not a competitor to BPE, just BPE trained in SentencePiece's raw-text machinery.

Quick check: what does SentencePiece BPE mean?

Two of these systems now dominate modern LLMs, so it is worth ending this section with the question every model builder actually faces: GPT-style byte-level BPE (OpenAI's tiktoken, encodings like cl100k_base and o200k_base) or SentencePiece? The honest answer is a tug-of-war between multilingual purity and strict structural control for math and code. Following are the considerations:

1. Regex pre-splitting buys structural control. SentencePiece feeds raw text straight into the merger, so pure BPE will fuse anything frequent: a comma onto a word (, The), or a whole year like 1999 or 2024 into one lump. GPT-style tokenizers run a strict regex first (the pre-tokenization step we dissect in the next section) that forbids letters, punctuation, and digits from merging across kinds and caps numbers at one- to three-digit chunks. That consistency is a real win for math and code, where the model learns digits as steady building blocks instead of a chaotic mix of lone digits and giant clumps.

2. Bytes guarantee zero unknowns. Traditional SentencePiece starts from the Unicode characters it saw in training, so a rare emoji or an unfamiliar script can fall through to an <unk> token and the model goes blind. (Modern SentencePiece can switch on byte fallback, but it was bolted on later.) GPT-style BPE converts everything to UTF-8 bytes first, with a base of exactly 256 byte values. So, as we saw down in the bytes section, it is impossible to hit an out-of-vocabulary character.

3. Raw text stays the most multilingual and self-contained. That same English-friendly regex is the catch: its rules about spaces and ASCII punctuation quietly assume Western-style text, so for a deeply multilingual model (Google's Aya, the early Gemma tokenizers) SentencePiece is the safer bet because it imposes no such rules. It also ships as a single self-contained .model file, so an inference server reproduces the exact tokenization without having to re-match a complex regex.

So which wins? For years SentencePiece was the open-source default: Llama 1 and 2, T5, and Mistral v1 and v2 all shipped it. But once coding and reasoning became the headline benchmarks, the math-and-code edge of regex pre-splitting began to outweigh its slight English bias, and the field pivoted: Llama 3 and Mistral v3 (its tekken tokenizer) both dropped SentencePiece for tiktoken-style byte-level BPE. The lesson is not that one is better; it is that the tokenizer follows the job: reach for SentencePiece when broad multilingual coverage leads, and for GPT-style byte-level BPE when math, code, and structural control lead.

4. Build the tokenizers

Let's build the tokenizers, one rule at a time.

Now let's look at BPE. We already know every string can fall down to bytes, and the GPT-style version of byte-pair encoding (BPE) starts exactly there: from the 256 UTF-8 byte values. The general BPE idea is broader than bytes, though. Some BPE tokenizers start from characters or SentencePiece pieces and only fall back to bytes for rare characters. In all cases, the learning rule asks: which neighboring pair appears so often that it deserves a shortcut? Merge that pair, mint a new token ID, replace it everywhere, and count again.

BPE has a trainer and an encoder.

The trainer sees a training corpus, counts neighboring pairs, learns a merge list, and freezes the vocabulary. At inference time, the encoder later sees a new user prompt (text), starts from the same learned base pieces, and replays that frozen merge list in order. It does not recalculate which pairs are common in your one prompt.

Note, if we run BPE completely naively and it wastes vocabulary slots on near-duplicates like dog, dog., dog!, and dog?. Hence, GPT-2 first splits text with a regular expression: contractions, Unicode letters, Unicode numbers, punctuation, and whitespace become separate regions, and BPE only merges within a region. Newer GPT-style encodings keep the same split-then-merge idea and tune a few places:

  • Numbers are capped at three digits. Long numbers become steadier groups (1234567 → 123, 456, 7) instead of one-off lumpy number tokens.
  • Runs of whitespace are grouped. Code stops paying one token for every indentation space, which matters a lot for Python.
  • Contractions are matched case-insensitively. DON'T and don't are less likely to split in oddly different ways.
Regex bucket Example chunk Why it matters before BPE
Contractions We're → We + 're Common endings like 's, 're, and 't can be handled consistently.
Unicode letters dogs, Um, lunch The leading space often travels with the next word, which is why dog and dog can be different tokens.
Unicode numbers 350 Digits can be grouped deliberately instead of letting every long number become a one-off token.
Punctuation !, ,, ? Punctuation does not automatically fuse with every nearby word and waste vocabulary slots.
Whitespace      or line breaks Code indentation and trailing spaces can be treated as text, not thrown away.

Tiny caveat for later: some newer research tokenizers add a second stage that can merge across spaces or punctuation after the safer first pass. The learner idea is still the same: the boundary policy decides what the learning rule is allowed to consider.

Now let's finalize one tiny corpus for the rest of this section, so the differences come from the learning rule itself: hug ×10, pug ×5, pun ×12, bun ×4, hugs ×5. These letters are really bytes in the byte-level story. For plain ASCII, h, u, and g each occupy one byte, so I write the byte as the letter to keep the example readable.

top pair counts: ug = 20, pu = 17, un = 16, hu = 15, gs = 5 top single-piece counts: u = 36, g = 20, s = 5

BPE asks only, "which neighboring pair appears most?" u g appears 20 times, so the first merge is u + g -> ug. It does not ask whether u also appears in many other places. Raw count wins.

Quick check: on each step, byte-pair encoding looks across the whole corpus and…

WordPiece asks, "which pair is unusually attached?" This is the key intuition. A pair can be common just because its two parts are common. WordPiece discounts that by dividing by the individual frequencies:

Score(A, B) = count(AB) / (count(A) * count(B)) Score(u, g) = 20 / (36 * 20) = 1/36 Score(g, s) = 5 / (20 * 5) = 1/20 1/20 > 1/36, so WordPiece prefers g + s even though it appears less often

Exercise: predict the first merge

  1. Using only raw counts, which pair wins: ug = 20 or gs = 5?
  2. Using WordPiece's score, which pair wins: 1/36 or 1/20?
  3. Explain the difference in one sentence without using the formula.

Check yourself: BPE picks ug. WordPiece picks gs because g and s are more specifically tied to each other.

Imagine you only have a strict budget of 30,000 "slots" to store the most useful chunks of language. BPE fills those slots by acting like a pure data-compression tool: it simply counts which letters or chunks sit next to each other most often (like `t` and `h`) and blindly merges them to save space. It doesn't care if the combination is meaningful; it only cares that it is frequent.

WordPiece, on the other hand, acts more like a linguist trying to maximize the actual *meaning* of each slot. It penalizes pairs made of letters that are incredibly common everywhere else. While `t` and `h` are common together, they also appear constantly on their own in other words, so WordPiece views their pairing as statistical noise. However, because `q` almost never appears without `u`, WordPiece prioritizes merging them. It makes this choice to avoid wasting precious vocabulary slots on frequent accidents of spelling, ensuring the model learns distinct, meaningful building blocks instead. That example is why WordPiece is not just "BPE with a different tie-breaker." It is trying to avoid wasting a merge on pieces that are only frequent because their ingredients are frequent everywhere. That makes it a natural fit for BERT-style masked-token prediction: the model benefits from stable, distinctive pieces such as play + ##ing, not merely the most common neighboring letters.

SentencePiece (Unigram LM) is the trickly one. We have not covered all the math yet, we will do so in the next chapter. However, the key idea is that it uses a probabilistic approach to determine the best way to segment text. Do not picture a greedy merge. Picture a menu. The tokenizer begins with many candidate pieces, gives each piece a probability, then asks which menu can explain the whole corpus with the lowest total surprise - remember surprise from our previous chapter. A useful piece earns its spot; a redundant piece gets pruned away.

candidate routes for pug: [p][u][g], [pu][g], [p][ug], [pug] route cost = -log P(piece1) - log P(piece2) - ... training: estimate piece probabilities, prune weak pieces, repeat encoding: choose the lowest-cost route through the lattice

Quick check: when a SentencePiece Unigram model encodes a word, what is it choosing?

The dynamic programming part is just the efficient way to avoid trying every possible cut by brute force. The learner intuition is simpler: SentencePiece (Unigram LM) keeps a scored menu of plausible pieces, then finds the cheapest path through the word or sentence. Its big practical win is not "newer than WordPiece"; it is raw-text, language-independent tokenization with reversible spaces.

At the end, we have the final piece: decoding. Encoding turned text into token IDs; decoding just runs it backwards. The model emits IDs, the tokenizer looks each one up in its frozen vocabulary, and the pieces are glued back into text. This is where SentencePiece's visible space marker earns its keep: because the space was stored as a real symbol (▁), turning ▁new▁words back into new words is exact and reversible, with no guessing about where the spaces belonged.

About the lab below: this is a teaching toy for the learning rules, not a clone of GPT-4, BERT, or SentencePiece. Each algorithm really runs: BPE merges by raw frequency, WordPiece merges by the count(AB) / (count(A) × count(B)) score, and Unigram LM starts from a big candidate set and prunes low-probability pieces. To stay fast and readable it skips the production boundary-policy choices we just discussed: regex chunks, raw SentencePiece spaces, byte fallback, normalization, and special tokens.

5. Why models break: the tokenization quirks

Many "the model is dumb" moments are really tokenizer moments.

A startling number of an LLM's famous failures are not failures of the neural network at all. They are failures of tokenization, hiding in plain sight. You need to start by checking the tokens. The weirdness stops being mysterious and starts being almost obvious. Remember the bedrock fact - the model never sees your letters, it sees chunks - so anything that depends on the letters inside a chunk is hard for it, and anything that bloats the number of chunks is expensive for it.

Pick a case in the lab below and watch the token stream; sometimes the visible text and the tokenized input are far less alike than they look. But first, let's make a little field guide. Each quirk below is a real, documented behavior, and every one traces straight back to how the bytes got grouped.

A token tax is evidence, not destiny

Suppose two names are equally important in the real world, but the tokenizer learned one from many examples and barely saw the other. The common name might be one token while the rare name becomes five pieces. That difference does not prove unfair behavior by itself, but it is a signal worth inspecting.

cost(common name) = 1 token cost(rare name) = 5 tokens same sentence + rare name = 4 extra tokens of context budget

This is the quiet fairness lesson in tokenization: representation begins before the neural network. A model can only learn from the pieces its tokenizer gives it. If the tokenizer's training corpus is dominated by one language or domain, that language often gets cheaper, chunkier pieces, while other languages pay more tokens for the same amount of meaning.

Can't spell or reverse words A word like DefaultStyle is one fat token, so its letters are hidden. Ask how many L's it has, or to reverse it, and the model fumbles.
Shaky arithmetic Numbers chunk arbitrarily - 127 is one token, 677 is two - so digit-by-digit math has no steady digits to grab. Llama deliberately splits every digit.
Non-English is expensive An English-dominated vocabulary may keep common English words whole while splitting basic Spanish, Hindi, Korean, or Thai words into many pieces. More pieces means more context budget and training compute for the same message.
Python broke GPT-2 Older GPT-style tokenizers often paid for indentation one space at a time, bloating code until it ran out of context. Newer encodings group whitespace more deliberately.
Trailing whitespace trips it A space is normally glued to the front of the next word ( of), so a lone trailing space is out of distribution and worsens the completion.
Special tokens are a trap Strings like <|endoftext|> are single special tokens; slipping one into a prompt can halt or derail the model - a genuine attack surface.
SolidGoldMagikarp A Reddit username that earned its own token in the tokenizer's data but never appeared in the model's, so its embedding row stayed random. Mention it and the model babbles.
YAML beats JSON The same data costs fewer tokens in YAML than in JSON, so in a pay-per-token world YAML is literally cheaper to send.

6. Going deeper

Further reading.

Tokenization is small in code and large in consequence. If you read only one thing after this, make it Karpathy's tokenizer talk - this whole chapter is my attempt to walk you through that same ladder slowly enough that you can own it. For a textbook companion, Jurafsky and Martin's "Words and Tokens" chapter is the cleanest broad reference. The rest of these links move from the core subword algorithms to the libraries and inspection tools used in production systems.

  1. Karpathy, Let's build the GPT Tokenizer (video) The two-hour talk this chapter is built around - bytes, BPE, GPT-style regex pre-tokenization, and every quirk, built up from scratch.
  2. Jurafsky and Martin, Words and Tokens A textbook treatment of words, morphemes, Unicode/UTF-8, BPE training and encoding, regex pre-tokenization, and multilingual tokenization costs.
  3. IIT Madras tokenization lecture: subword motivation The word-level vs character-level tradeoff behind this section. Companion lectures cover BPE, WordPiece, and SentencePiece.
  4. Gage, A New Algorithm for Data Compression The original 1994 Byte Pair Encoding article, written as a general-purpose compression algorithm.
  5. Sennrich, Haddow, and Birch, Neural Machine Translation of Rare Words with Subword Units The influential paper that brought byte pair encoding to neural models.
  6. Wu et al., Google's Neural Machine Translation System The production-scale translation paper that explains wordpieces as a rare-word solution between characters and whole words.
  7. Devlin et al., BERT The BERT paper; it uses a 30,000-token WordPiece vocabulary and masked-token prediction.
  8. Radford et al., GPT-2 The source for byte-level BPE as the bridge between word-level efficiency and byte-level coverage.
  9. DeepSeek-AI, DeepSeek-V3 Technical Report A modern open-weight example of byte-level BPE with a large multilingual vocabulary and a modified pretokenizer.
  10. Qwen2.5 Technical Report A modern open-weight example of byte-level BPE with a unified 151,643-token regular vocabulary.
  11. Baichuan 2 Technical Report A clear example of BPE trained with SentencePiece, including digit splitting, whitespace-only tokens, and UTF-8 byte fallback.
  12. Kudo, Subword Regularization and Unigram Tokenization The primary source for the Unigram language-model tokenizer many readers meet through SentencePiece.
  13. Kudo and Richardson, SentencePiece Raw-text subword tokenization and detokenization for neural text processing.
  14. BERT Tokenization Code A readable implementation reference for BasicTokenizer and WordPieceTokenizer.
  15. Hugging Face Tokenizers Course A practical, example-driven tour of training and using modern tokenizers.
  16. tiktoken OpenAI's fast byte-level BPE tokenizer, useful for counting real token costs.
  17. minbpe Karpathy's minimal, readable BPE implementation - the code companion to the video above.
  18. Tiktokenizer Paste any text and watch how different models slice it into tokens, live in the browser.
