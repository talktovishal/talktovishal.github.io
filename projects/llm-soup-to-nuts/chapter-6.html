Learning by doing

Training and alignment: how a base model becomes useful.

In Chapter 5 you ran a finished Transformer. This chapter answers the question that naturally follows: where did that model come from, and why is a raw pretrained model still not the same thing as a helpful assistant?

The path has stages. First comes pretraining, where the model predicts the next token across enormous text streams. Then comes post-training: supervised instruction tuning, preference alignment, and sometimes extra thinking at inference time. The same next-token machine is underneath, but the training signal changes what behavior we ask it to learn.

This rewrite is grounded in the open Speech and Language Processing lectures on Large Language Models and Post-training. I am using those lectures as the coverage map, while keeping the course pattern: intuition, a small worked calculation, symbols, and a live panel.

Pretrain Predict the next token with teacher forcing and cross-entropy.
Choose data Filter, deduplicate, document, and notice what the corpus teaches.
Scale Balance parameters, tokens, compute, and memory.
Align Use instructions and preferences to make the model more helpful.
Think at test time Spend extra inference work when a task needs step-by-step reasoning.

Before we dive in, a few needed prerequisites are in order. This chapter stitches together the pieces you have already built. Use the links as quick refreshers if one of them feels rusty.

  • Next-token prediction. A language model assigns probabilities to possible next tokens.
  • Tokenization. Text becomes token IDs before the model ever sees it.
  • Softmax, cross-entropy, and gradients. Training lowers surprise by nudging weights.
  • Transformer forward pass. Tokens become vectors, blocks update them, and the LM head makes logits.

What you do not need yet: production distributed-systems detail. We will count memory and compute enough to understand the pressure, then leave cluster engineering for deeper references.

1. Training stages

A model is trained more than once.

A base model starts as random numbers. Pretraining teaches it broad language, facts, styles, and patterns by asking one simple question again and again: given the previous tokens, what comes next? That makes it a strong text predictor, but not automatically a good assistant.

Post-training changes the behavior we care about. Instruction tuning shows the model examples of requests and good responses. Preference alignment shows it pairs of responses and asks it to prefer the one humans judge as better. Test-time compute is different again: it spends extra work during inference, often by encouraging intermediate reasoning steps.

How to read a model card without mixing things up

architecture vs weights

Model cards often put several kinds of facts next to each other. Keep them separate and the rest of this chapter becomes easier to place.

Architecture The recipe: decoder-only Transformer, attention, FFN, norms, RoPE, dense or MoE.
Configuration The shape: layers, width, heads, vocabulary size, context length, dense vs active parameters.
Weights The learned numbers after training. Same architecture, different weights, different model.
Training stage Base, instruction-tuned, preference-aligned, or a system spending extra test-time compute.

A 70B base model and a 70B chat model may share the same broad architecture. The behavior difference usually comes from weights and post-training, not a brand-new forward pass.

2. Pretraining loss

The text itself supplies the labels.

Pretraining is self-supervised. We do not need humans to label every token because the next token is already sitting in the text. At position \(t\), the model reads the true prefix \(w_1,\ldots,w_t\), predicts a distribution over the vocabulary, and pays loss on the real next token \(w_{t+1}\).

During training we use teacher forcing: even if the model would have guessed the wrong token at one position, the next training position still receives the true history from the corpus. That keeps the learning signal clean and parallelizable across the whole sequence.

One tiny batch by hand

Suppose the corpus slice is So long and thanks for all. The model predicts after each prefix. If it assigns these probabilities to the correct next tokens, the batch loss is the average negative log probability:

P(long | So) = 0.50, loss = -ln(0.50) = 0.69 P(and | So long) = 0.40, loss = -ln(0.40) = 0.92 P(thanks | So long and) = 0.25, loss = -ln(0.25) = 1.39 average loss = (0.69 + 0.92 + 1.39) / 3 = 1.00 nats perplexity = exp(1.00) = 2.7 effective next-token choices

The same training step, in symbols

Cross-entropy over a one-hot target collapses to the negative log probability of the true next token. Average that over a sequence or batch:

$$L_t = -\sum_{w \in V} y_t[w]\log \hat{y}_t[w] = -\log \hat{y}_t[w_{t+1}]$$ $$L_{\text{batch}} = \frac{1}{T}\sum_{t=1}^{T} -\log \hat{y}_t[w_{t+1}] \qquad \text{perplexity}=e^{L_{\text{batch}}}$$
  • \(V\): the vocabulary of tokens.
  • \(y_t\): the one-hot target, 1 for the true next token and 0 elsewhere.
  • \(\hat{y}_t\): the model's predicted distribution from the LM head.
  • \(T\): the number of predicted positions in the batch.

3. Training data

The corpus is a curriculum, whether we admit it or not.

Large language models are trained on web crawls plus curated sources such as books, Wikipedia, academic papers, code, and dialogue. The source lecture names open corpora like Common Crawl derivatives, C4, The Pile, and Dolma to make the scale concrete. The exact mixture matters because the model learns from frequency, not from what we wish were important.

A data pipeline filters for language and quality, removes boilerplate and duplicates, tries to reduce personal information and unsafe content, checks for benchmark contamination, and then tokenizes the cleaned text. Every stage is a choice. A strict toxicity filter may reduce some harmful generations while also removing dialectal text unfairly. Deduplication can stop one copied page from overpowering thousands of unique pages. Copyright, data consent, privacy, and geographic skew are not side notes; they shape what the base model knows.

Deduplication by hand

If one page appears 100 times beside 900 single-copy pages, it is 10% of the examples before deduplication. After exact deduplication, it is one page out of 901:

before dedup: 100 / (100 + 900) = 10% after dedup: 1 / (1 + 900) = 0.11% mixture example: 100B tokens * 20% code = 20B code tokens

4. Scaling and systems

Spend a fixed compute budget without starving the model.

Scaling laws say that loss falls predictably as we add parameters, data, and compute. The trap is that training compute is roughly \(C \approx 6ND\), where \(N\) is parameter count and \(D\) is training tokens. With a fixed budget, a bigger model must see fewer tokens. If it sees too little data, it is undertrained.

For a dense decoder Transformer, a rough parameter estimate is \(N \approx 12\,n_{\text{layer}}d^2\). A GPT-3-scale shape with 96 layers and width 12,288 lands around 175B parameters. During training, examples are often packed into full context windows with end-of-text separators, and batches are counted in tokens, not just documents.

That estimate is for a dense model, where every token uses the same feed-forward weights. Mixture-of-Experts models complicate model-card reading: they may report a large total parameter count, but a router activates only a few expert FFNs for each token. When you compare compute or inference cost, look for active parameters, not only total parameters.

Compute by hand

If you train a 1B-parameter model on 20B tokens, the run costs about \(1.2 \times 10^{20}\) FLOPs by the rough dense-Transformer rule:

C = 6 * 1B parameters * 20B tokens = 1.2e20 FLOPs same C with 10B parameters gives D = C / (6N) = 2B tokens tokens per parameter: 2B / 10B = 0.2, far below a Chinchilla-style balance

Chinchilla's rule of thumb: for a dense model, compute-optimal training spends roughly 20 tokens per parameter - so that 10B model really wants about 200B tokens, not 2B. Several early giant models were badly undertrained: too many parameters fed too little text, which is exactly the imbalance the slider lets you feel below.

The scaling pressure, in symbols

The constants vary across architectures and datasets, but the lesson is stable: loss has separate terms for model size and data size, while compute ties them together.

$$L(N,D) \approx E + \frac{A}{N^\alpha} + \frac{B}{D^\beta} \qquad C \approx 6ND$$ $$N \approx 12\,n_{\text{layer}}d^2$$
  • \(N\): non-embedding parameter count.
  • \(D\): training tokens.
  • \(C\): training compute in FLOPs.
  • \(d\): model width, the size of each residual stream.

Why one GPU is never enough

Training memory is dominated not only by weights, but also gradients, optimizer state, master weights, activations, and communication buffers. Data parallelism splits batches. Tensor and pipeline parallelism split model computation. ZeRO and FSDP shard weights, gradients, and optimizer state so one device does not carry the whole model.

5. Evaluation and safety

Low loss is useful, but it is not the whole report card.

Perplexity measures how surprised the model is by held-out text. Lower is better, and it connects directly to the pretraining loss. But perplexity is sensitive to tokenization, so it is most meaningful when comparing models with the same tokenizer.

We also evaluate downstream tasks such as MMLU-style multiple-choice questions, reasoning tasks, translation, summarization, latency, energy, fairness, and robustness. Public benchmarks bring a hard warning: data contamination. If test questions leaked into training, the score can overstate real ability.

Long-context evaluation needs its own care. A huge context window does not prove the model uses every position equally well. Tests like key-value retrieval and multi-document question answering can reveal a lost-in-the-middle pattern, where evidence near the middle of the prompt is harder for the model to use than evidence near the start or end.

Safety is not separate from evaluation. A model can hallucinate, agree with a false premise, expose memorized private data, produce toxic or stereotyped text, or give dangerous advice. These are not just bad vibes; they are failure modes of the training objective and data. Post-training tries to reduce them, but evaluation has to keep checking.

6. SFT and LoRA

Instruction tuning teaches the model what a request means.

A pretrained model knows how text tends to continue. That is not the same as knowing that Translate to French: The small dog is a request to translate, not a prefix to continue in English. Instruction tuning, also called SFT, continues training on instruction-response examples using the same next-token cross-entropy loss.

Instruction data can be written by people, converted from existing supervised datasets with templates, created from detailed task guidelines, or generated synthetically and reviewed. The goal is not one task; it is learning the habit of following instructions across tasks and formats.

Full fine-tuning updates all parameters. LoRA is a parameter-efficient alternative: freeze a large matrix \(W\), train two small matrices \(A\) and \(B\), and add their low-rank product into the forward pass.

Same loss, smaller update

SFT uses the same language-modeling loss as pretraining, but the text is curated request-and-response data. LoRA changes how many parameters move:

$$L_{\text{SFT}} = \frac{1}{T}\sum_{t=1}^{T} -\log \hat{y}_t[w_{t+1}]$$ $$h = xW + xAB \qquad r \ll \min(d,k)$$
  • \(W\): a frozen pretrained weight matrix.
  • \(A,B\): small trainable matrices whose product has the same shape as \(W\).
  • \(r\): the low rank, much smaller than the original dimensions.

7. Preference alignment

Preferences turn "better response" into a training signal.

After SFT, a model can still be unhelpful, overconfident, unsafe, or correct but awkward. Preference alignment starts from examples like: for this prompt, response A was preferred over response B. The annotator does not have to write the perfect answer; they only need to choose or rank outputs.

The Bradley-Terry model gives the bridge from preferences to probabilities. If the reward score for the chosen response is much higher than the rejected response, the model should assign a high probability to the preference. If the scores are close, the preference is uncertain.

Preference by hand

Say a reward model scores the chosen answer 8.0 and the rejected one 5.0. The only thing that matters is the gap - here 8.0 - 5.0 = 3.0. Squeeze that gap through the sigmoid and you get the probability that a person prefers the chosen answer:

gap = r(chosen) - r(rejected) = 8.0 - 5.0 = 3.0 P(prefer chosen) = sigmoid(3.0) = 1 / (1 + e^-3.0) ~= 0.95

That is the whole reason a sigmoid shows up: a big positive gap means "almost certainly the better answer" (near 1), a gap of zero means "a coin flip" (0.5), and a negative gap means the scores are backwards (below 0.5). Training just pushes the gap on real preference pairs to be positive. Drag the reward gap slider below and watch this same curve move.

Preference math, in symbols

Let \(o_w\) be the winning response and \(o_l\) the losing response for a prompt \(x\). A reward model learns a score \(r(x,o)\), then alignment methods nudge the policy toward preferred outputs without drifting too far from a reference model.

$$P(o_w \succ o_l \mid x) = \sigma(r(x,o_w) - r(x,o_l))$$ $$L_{\text{reward}} = -\log \sigma(r(x,o_w) - r(x,o_l))$$ $$\pi^* = \arg\max_{\pi} \mathbb{E}_{x,o \sim \pi}[r(x,o) - \beta \log \frac{\pi(o|x)}{\pi_{\text{ref}}(o|x)}]$$ $$L_{\text{DPO}} = -\log \sigma\!\left(\beta\log\frac{\pi_\theta(o_w|x)}{\pi_{\text{ref}}(o_w|x)} - \beta\log\frac{\pi_\theta(o_l|x)}{\pi_{\text{ref}}(o_l|x)}\right)$$
  • \(\sigma\): logistic sigmoid, turning a score difference into a probability.
  • \(\pi_\theta\): the model being trained.
  • \(\pi_{\text{ref}}\): the reference model we do not want to drift too far from.
  • \(\beta\): the strength of the stay-close penalty.

Read these four lines as one story. The first scores a preference (the sigmoid of the reward gap you just computed by hand). The second trains a reward model to predict those preferences. The third is the RLHF goal: tune the policy to chase high reward while the \(\beta\) term penalises drifting away from the reference model. The last, DPO, is the clever shortcut - it reaches the same goal directly from the preference pairs, with no separate reward model to train at all.

8. Test-time compute

Some work happens after training is over.

Pretraining, SFT, and preference alignment all change parameters. Test-time compute does not. It spends extra inference work when answering a prompt. The classic example is chain-of-thought prompting: demonstrations include reasoning steps, so the model is more likely to produce intermediate steps before the final answer.

The tradeoff is simple. More generated reasoning tokens can improve difficult reasoning tasks, but they cost time and tokens. The point is not that every response should be long. It is that inference-time work is another lever, after the weights are already trained.
