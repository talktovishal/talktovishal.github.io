Learning by doing

Pretraining: turn a pile of text into a base model.

In Chapter 5 you ran a finished Transformer. This chapter is about where that model comes from. Pretraining is the long, expensive first phase where a fresh network reads an enormous amount of text and does one simple thing billions of times: predict the next token, check how surprised it was, and nudge its weights. The same four-beat loop from Chapter 3, just very, very large.

Three decisions shape the result: the objective the model trains on, the data it reads, and how much compute you spend. Get the balance right and the model keeps improving in a predictable way. This chapter lets you play with each decision and watch the tradeoffs that real labs wrestle with.

Guiding question: if you had a fixed budget of computer time, would you spend it on a bigger model or on more data? The answer is not obvious, and getting it wrong wastes millions of dollars.

1. Pretraining objectives

What, exactly, is the model trying to predict?

An objective is the game the model plays during training. The dominant one for chat models is next-token prediction, also called causal language modeling: read the tokens so far, predict the next one. It matches Chapter 1's question exactly, and because it only ever looks backward, the trained model can generate text left to right.

A different objective, masked language modeling, hides a few tokens and asks the model to fill them in using context from both sides. That builds rich understanding of a whole sentence, which is why it powers search and classification models, but it does not directly give you a generator. Switch between the two below and watch which tokens become targets and what context each one is allowed to use.

Under the hood, next-token pretraining is still the cross-entropy story from Chapter 3. At position t, the model reads the earlier tokens x_<t, assigns a probability to the real next token x_t, and pays surprise -ln p(x_t | x_<t). Average that over billions of positions and you have the training loss.

One next-token loss by hand

Suppose the context is the cat sat on the and the real next token is mat. If the model assigns mat a probability of 0.40, the loss for this one position is:

loss = -ln p(real next token | previous tokens) loss = -ln 0.40 = 0.92 nats if average loss = 1.20 nats, perplexity = exp(1.20) = 3.3 tokens

Perplexity is back: it is the model's average uncertainty written as an effective number of next-token choices.

2. Training data

Most of pretraining is really data preparation.

A base model is a mirror of its training data, so labs spend enormous effort on what goes in. Raw web text is huge but messy. It gets filtered for language and quality, deduplicated so repeated pages do not dominate, stripped of personal information and unsafe content, and checked so that test benchmarks do not leak in. Only then is it tokenized into the stream the model actually trains on.

The choices here are not just technical. Licensing and copyright, removing personal data, filtering toxic content, and avoiding benchmark contamination are all decisions with real consequences, and they are increasingly documented in dataset and model cards.

Data preparation by hand

Deduplication is not just storage cleanup; it changes what the model is trained to pay attention to. If one page appears 100 times beside 900 single-copy pages, that one page is 10% of the examples before dedup. After exact deduplication, it is just one page out of 901.

before dedup: repeated page share = 100 / (100 + 900) = 10% after dedup: repeated page share = 1 / (1 + 900) = 0.11% mixture example: 100B tokens * 20% code = 20B code tokens

The model does not know what "important" means. It sees frequency, mixture weights, and loss. Data preparation is where humans decide what evidence gets repeated and what gets removed.

3. Scaling laws

Spend a fixed compute budget: bigger model or more data?

Here is one of the most useful results in modern AI: model loss falls in a smooth, predictable way as you add parameters and data. A good approximation is loss = E + A / N^α + B / D^β, where N is the number of parameters and D is the number of training tokens. Training compute is roughly C ≈ 6 · N · D.

That last equation is the catch. For a fixed compute budget you cannot grow both N and D at once: a bigger model must train on less data, and vice versa. Pick a budget, then slide the model size along it. The curve shows the predicted loss, and the marked point is the compute-optimal balance, the lesson made famous by the Chinchilla result: most early large models were too big and badly undertrained.

Compute by hand

A rough training-compute rule for dense decoder Transformers is C ≈ 6ND. It is not exact engineering accounting, but it is good enough to feel the tradeoff. If you train a 1B-parameter model on 20B tokens, the run costs about 1.2 * 10^20 FLOPs.

C = 6 * 1B parameters * 20B tokens = 1.2e20 FLOPs same C with 10B parameters gives D = C / (6N) = 2B tokens tokens per parameter: 2B / 10B = 0.2, far below the ~20 Chinchilla-style target

Bigger is not automatically better. With the same compute, a much bigger model may see too little data and stay undertrained.

4. Distributed training

Why one GPU is never enough.

Training memory is dominated not by the weights but by everything you keep around them. For each parameter you typically store fp16 weights, fp16 gradients, and the optimizer's fp32 momentum, variance, and master copy, roughly 16 bytes per parameter before you even count activations. Slide the model size below and watch the total blow past a single GPU almost immediately.

That is why large runs spread the work across many GPUs. Data parallelism copies the model and splits the batch. Tensor and pipeline parallelism split a single model across devices. ZeRO and FSDP shard the weights, gradients, and optimizer state so no single GPU holds all of it. The sharding slider shows the effect on per-GPU memory directly.

The 16 bytes estimate is only the training state for the parameters. Real training also needs activation memory for the forward pass, temporary communication buffers, and room for batches. That is why "fits the optimizer state" is necessary but not sufficient.

Training memory by hand

With Adam-style training, a single parameter is not just one fp16 weight. You also keep its gradient, two optimizer statistics, and often an fp32 master copy. That rough accounting gives the 16-byte rule used below.

bytes per parameter = 2 weight + 2 gradient + 8 Adam state + 4 master weight = 16 7B parameters * 16 bytes = 112GB before activations 70B parameters * 16 bytes = 1,120GB = 1.12TB before activations

Sharding does not make the model smaller; it divides the state across devices so each GPU carries only part of the burden.

5. Going deeper

Further reading.

These references cover the objectives, the data, the scaling laws, and the systems that make pretraining possible at scale.

  1. Kaplan et al., Scaling Laws for Neural Language Models The first clear power-law relationship between loss, model size, data, and compute.
  2. Hoffmann et al., Training Compute-Optimal Large Language Models (Chinchilla) The result that most large models were undertrained, and how to balance N and D.
  3. Gao et al., The Pile A widely used, openly documented pretraining dataset.
  4. Penedo et al., The RefinedWeb Dataset Large-scale filtering and deduplication of web text for pretraining.
  5. Rajbhandari et al., ZeRO Memory optimizations that shard optimizer state, gradients, and weights.
  6. Shoeybi et al., Megatron-LM Tensor and pipeline parallelism for training very large Transformers.
  7. Hugging Face, Multi-GPU Training Methods A practical guide to data, tensor, and pipeline parallelism.
  8. Raffel et al., T5 A careful study of pretraining objectives and transfer with a unified text-to-text setup.
  9. Alammar, The Illustrated BERT A visual explanation of masked-language pretraining and contextual embeddings.
